# Security Policy

## Supported versions

The maintainers focus security fixes on the latest published release line.

## Reporting a vulnerability

- Do not open a public issue for credential leaks, insecure defaults, or vulnerabilities involving real payment flows.
- Use GitHub’s private vulnerability reporting flow if it is enabled for the repository.
- If private reporting is unavailable, contact the maintainers directly and include a minimal reproduction with all secrets removed.

## What to include

- A clear description of the issue
- Impact and affected API flows
- Reproduction steps
- Any suggested mitigation or patch direction

## Sensitive data

Never include real consumer keys, consumer secrets, initiator passwords, security credentials, or customer identifiers in reports.
